The mental model that makes CBVs click

You never call these classes. You set attributes, and inherited code you never see reads them by name. That's why page_size can't be renamed to pageSize — the parent class goes looking for that exact string.

Express is "write the function, it runs." Django CBVs are "fill in the config, the framework runs its function." Inversion of control is the whole adjustment.

python manage.py makemigrations

vs Express There's no Django equivalent of scattering routes and schema wherever you like. The tradeoff: you lose layout freedom, you gain the fact that every Django project on earth looks like this one. Migrations are auto-generated from models — closer to Prisma than to hand-written Knex.
New app checklist startapp → add to INSTALLED_APPS in settings.py → write models → makemigrations → migrate → include its urls in the root urls.py. Forgetting INSTALLED_APPS is the classic silent failure.

ORM & querysets

The single most important idea: a queryset is a recipe, not a result.

BUILD.filter().order_by()Nothing runs. Chainable.
→
SLICEqs[0:10]Still nothing — adds LIMIT
→
EVALUATEfor x in qs / list(qs)NOW the SQL runs

That laziness is why pagination is cheap: slicing a queryset becomes LIMIT/OFFSET in SQL. Rows never enter Python.

# chain freely — one query at the end, not four
items = (MenuItem.objects
    .filter(is_available=True)
    .exclude(price__gt=500)
    .select_related('category')
    .order_by('name'))

.get() raising is why item_detail in this repo wraps it in try/except. .filter(...).first() returns None instead — that's what the new MenuItemDetailView.get_object() uses, so no exception handling is needed.

vs Mongoose await Model.find() executes immediately; Django's .filter() doesn't. No await anywhere — Django is synchronous by default, and that's fine. select_related is .populate(), but it does a real SQL JOIN.
N+1, the one that bites Serializing 10 items with category_name fires 1 query + 10 more for categories. .select_related('category') collapses it to one JOIN. Use it for FK / OneToOne; use .prefetch_related() for reverse FK and ManyToMany.

Two jobs in one class: shape the JSON, and validate what comes in.

class MenuItemSerializer(serializers.ModelSerializer):
    category_name = serializers.CharField(source='category.name', read_only=True)

    class Meta:
        model  = MenuItem
        fields = ('id', 'name', 'price', 'category', 'category_name')
        read_only_fields = ('id',)

    def validate_price(self, value):      # validate_<field> — auto-discovered
        if value <= 0:
            raise serializers.ValidationError("Price must be greater than zero")
        return value
PatternDoes
Serializer(obj)read one → .data
Serializer(qs, many=True)read a list

Serializer(obj, data=…, partial=True)partial update (PATCH)

validate_<field> checks one field; a validate(self, attrs) method checks the whole object when a rule spans two fields.

vs Express Zod or Joi, but bidirectional — the same class that validates input also formats output, and ModelSerializer infers the fields from the model instead of you redeclaring them.

Pagination

A config class the view hands the queryset to. Four attributes, read by name.

class MenuPagination(PageNumberPagination):
    page_size = 10                   # default per page
    page_size_query_param = 'size'   # client override: ?size=50
    max_page_size = 100              # cap it

Two ways to use it

# generic view — set one attribute, the rest is inherited
class MenuItemListView(generics.ListAPIView):
    pagination_class = MenuPagination

# APIView — the same work, done by hand (this repo does this)
paginator  = MenuPagination()
page       = paginator.paginate_queryset(items, request, view=self)
serializer = MenuItemSerializer(page, many=True)
return paginator.get_paginated_response(serializer.data)

paginate_queryset reads ?page and ?size, clamps size against max_page_size, then slices → LIMIT 10 OFFSET 20. The response envelope is always:

{ "count": 137, "next": "…?page=3", "previous": "…?page=1", "results": [ … ] }
ClassQueryUse when
PageNumberPagination?page=2numbered pages
LimitOffsetPagination?limit=&offset=client controls the window
CursorPagination?cursor=…infinite scroll, huge tables
Always order the queryset LIMIT/OFFSET without ORDER BY lets the database return rows in any order, and it can differ between the page-1 and page-2 queries — items duplicate across pages while others never appear. MenuItem has no Meta.ordering and a random UUID primary key, so there's no accidental insertion-order to save you.
Set a project-wide default 
# settings.py
REST_FRAMEWORK = {
    'DEFAULT_PAGINATION_CLASS': 'rest_framework.pagination.PageNumberPagination',
    'PAGE_SIZE': 20,
}

Setting DEFAULT_PAGINATION_CLASS without PAGE_SIZE silently disables pagination. Not set in this repo yet — so OrderViewSet currently returns every order in one response.

Two styles — same trade as the view ladder

Lines~25~7

The declarative version the lecturer showed:

class MenuItemListView(generics.ListAPIView):
    filter_backends = [filters.SearchFilter, filters.OrderingFilter]
    search_fields   = ['name', 'description', 'category__name']
    ordering_fields = ['name', 'price', 'created_at']  # whitelist!
    ordering        = ['name']                       # default sort

The by-hand version — the whitelist gate is the whole lesson:

ordering = request.query_params.get('ordering', 'name')
if ordering.lstrip('-') in {'name', 'price', 'created_at'}:  # -price → 'price'
    items = items.order_by(ordering)                       # leading - = DESC
else:
    items = items.order_by('name')                        # ignore junk input
Never pass ?ordering= straight into order_by() Raw user input lets a client sort by any field or FK traversal (?ordering=category__id), or crash the query on a typo. Whitelist it — that's exactly what the lecturer's # whitelist! comment guards. The declarative backend does the same check internally; by hand you spell it out.
Why this repo stayed by-hand SearchFilter / OrderingFilter give ?search= and ?ordering= free, but not the custom ?category= / ?available= exact filters — those need a third backend (DjangoFilterBackend, pip install django-filter). Writing get() keeps all four filters in one place, no extra dependency.

Caching

Store a response and re-serve it without touching the DB. cache_page is a Django tool (not DRF), written for function views.

Decorate the handler your base class actually calls

Base classList handlerDecorate
APIView (this repo)get()get
ListAPIView / ModelViewSetlist()list
from django.utils.decorators import method_decorator
from django.views.decorators.cache import cache_page

class MenuItemListView(APIView):
    @method_decorator(cache_page(60 * 5))   # 5 min TTL
    def get(self, request):                # NOT list() — this is an APIView
        …

cache_page takes (request); a method gets (self, request). method_decorator bridges that gap — like wrapping Express middleware to sit on a class method. It keys on the full URL incl. query string, so every filter combo caches on its own:

/menu/items/?search=iphone&ordering=-price  # entry A
/menu/items/?category=Breads                 # entry B (independent)
/menu/items/?search=iphone&ordering=-price  # HIT — no DB, no query
No auto-invalidation cache_page never clears itself. POST a new item and the cached list keeps serving the old set until the TTL expires. Fine for a slow-changing public menu; a bug for fast-moving data. Only GET/HEAD are cached — post() is untouched.
It caches globally — no concept of "user" One response is stored for everyone. Perfect for a public menu; on a per-user endpoint it leaks — user A gets user B's cached data. Per-user caching needs a manual key or vary_on_headers.
Where the cache actually lives — CACHES 
# settings.py — default if you set nothing:
CACHES = {'default': {'BACKEND':
    'django.core.cache.backends.locmem.LocMemCache'}}  # in-process, wiped on restart

# real deployments point it at a shared store:
CACHES = {'default': {
    'BACKEND': 'django.core.cache.backends.redis.RedisCache',
    'LOCATION': 'redis://127.0.0.1:6379'}}

LocMemCache is per-process — with multiple workers each has its own copy, so a hit rate looks random. Redis/Memcached give one shared cache across workers.

vs Express cache_page ≈ apcache/res.set('Cache-Control') plus a server-side store — but Django keeps the rendered response, so a hit skips your view entirely, the way a reverse proxy would.

URLs & routing

Three wiring styles. Using the wrong one is the most common early error.

urlpatterns = [
    path('list/', views.menu_list),                      # function → bare
    path('api/items/', views.MenuItemListView.as_view()), # class → .as_view()
]

router.register('', OrderViewSet, basename='order')      # viewset → router
Why .as_view() Django's resolver can only call functions. as_view() converts the class into one. A ViewSet has no get() at all — the router maps list→GET, create→POST, etc. That's the entire reason routers exist.

Path converters

The converter validates before your view runs — a malformed UUID gives a clean 404 at routing time rather than a 500 inside the query.

vs Express path() is app.get() minus the verb — Django matches URL first, then the view decides the method. include() is app.use('/prefix', router). No :param — Django types the params for you.

Adding a unique field to a table that already has rows

The trap that broke orders/0006. AddField evaluates default=uuid.uuid4 once and writes that one value to every existing row — so with 2+ rows, unique=True fails immediately:

sqlite3.IntegrityError: UNIQUE constraint failed: new__orders_order.public_id

It isn't fixable by retrying, because "one default for every row" and "all values distinct" can't both be true. Split it into three operations in the same file:

def backfill_public_ids(apps, schema_editor):
    Order = apps.get_model('orders', 'Order')   # NOT a real import
    for order in Order.objects.all():
        order.public_id = uuid.uuid4()
        order.save(update_fields=['public_id'])

operations = [
    # 1. add it WITHOUT unique — duplicates allowed for now
    migrations.AddField('order', 'public_id',
        models.UUIDField(default=uuid.uuid4, editable=False)),

    # 2. backfill one distinct value per row
    migrations.RunPython(backfill_public_ids, migrations.RunPython.noop),

    # 3. NOW apply the constraint
    migrations.AlterField('order', 'public_id',
        models.UUIDField(default=uuid.uuid4, unique=True, db_index=True, editable=False)),
]

Same shape for any "backfill then constrain" change — adding null=False to a populated column, splitting a name field, renaming a status value.

Always apps.get_model() in a data migration Never from orders.models import Order. A migration must see the model as it was at that point in history. Import the live model and this file breaks retroactively the day you delete a field — an old migration would suddenly reference a column that no longer exists.
When editing a migration by hand is safe

Only while it's still [ ] unapplied — as it was here. Once a migration has run anywhere it can't be edited, because Django records it as done by name and never re-reads it. At that point you write a new corrective migration instead.

Rule of thumb: unapplied and local → edit freely. Applied or pushed → add, never edit.

vs Prisma / Knex makemigrations is prisma migrate dev — diffed from your schema, not hand-written. But Django migrations are plain Python you're expected to open and edit, and RunPython gives you a data migration in the same file as the schema change.

One PROTECT among the CASCADEs Deleting a Category deletes its menu items. But deleting a MenuItem that appears on any order raises ProtectedError — because OrderItem.menu_item uses PROTECT. That's correct: it stops a menu edit from silently rewriting order history.

Three inheritance styles — you're using all of them

StyleDeclared byTablesHere
Abstractabstract = Truenone — columns copied into childrenTimestampedModel
Multi-tablesubclass a concrete modelown table + implicit 1–1Food, Beverage

Rule of thumb: abstract for shared columns, proxy for shared columns with different behaviour, multi-table when a subtype genuinely has extra columns. Multi-table is the one to be careful with — every read is a JOIN.

vs Mongoose No embedding here. Mongo would nest items inside the order document; a relational schema makes OrderItem its own table so a menu item can be queried across all orders. related_name is what .populate() gives you, but declared on the model.
